SEC·04

The improvement loop

propose-only · human reviews every diff

Classify failures

Failed A/B runs cluster by domain + failure mode (NO_TOOL_CALL, WRONG_TOOL, MULTI_STEP_DROPOUT…), each cluster pointed at a target artifact.

Bandit picks strategy

Thompson-sampling over arms like add-edge-case, tighten-specificity, push-tool-action per layer. Crashes count as losses.

ReflACT trainer

Iterative edit-propose-score on a train/selection split; held-out test split guards against overfitting. Targets one artifact: SKILL.md body or one harness key.

Propose only

Winners land as *_proposed.md / .yaml + a report. The optimizer never touches live skills or config. Archive keeps every candidate with lineage.

Human review

Diff against live artifact. Drop hallucinated tools, off-topic additions, repetition. ~50% of proposal content survives review — by design.

Apply + re-gate

Reviewed edit applied, full A/B re-run, tiered gate decides. Yesterday: three failing skills → all PASS in one loop day.

↻ new failures feed the next classification — the loop never claims "done"
Guardrail

--cluster-timeout

SIGALRM watchdog per cluster; a stuck network call once froze a batch 2.5h — now it's skipped and the batch continues.

Guardrail

--max-rollout-calls

Pre-flight cost estimate across all clusters; over-budget batches refuse to start. Fired twice in production, saved ~4,000 calls.

Guardrail

--heartbeat

Liveness line every N seconds so a silent stall is visible in seconds, not hours.

Guardrail

Crash accounting

A crashed/timed-out run records a loss on its bandit arm — strategies that keep blowing up get deprioritized automatically.

SEC·05

CI/CD — the referee

github actions · verified live on PR #1

tests.yml — playground

on: push / pull_request
  1. Fresh ubuntu runner, pip install (requirements verified on clean machine)
  2. 284-test suite — optimizer, verifiers, taskgen, gate, mock server, archive, bandit
  3. No API keys needed — everything heavy is stubbed

eval.yml — skill eval gate

on: pull_request touching skills/**
  1. Validate skill format + security scan: tool references vs approved_tools.json
  2. Cross-repo checkout of the eval platform (owner-parameterized)
  3. Boot mock MCP, health-polled
  4. 5-trial A/B on the changed skill's bank (50 tasks ≈ 500 calls ≈ $0.03 ≈ 4 min)
  5. Tiered verdict posted as PR comment — BLOCK means don't merge

Division of labor: local = build (task generation, optimizer runs, fix iterations) · CI = judge (every skill-touching PR gets a verdict before a human merges).

SEC·07

Roadmap — the third layer

skill text → harness → weights

Weights via API fine-tuning

Harvest passing trajectories from trajectory.db → SFT JSONL → fine-tune gpt-4o-mini via API → A/B the returned ft: model id with the existing gate. New bandit arm: weights:sft. Targets the behavioral failure class ("asks instead of acts") that survives prompt surgery.

Detection-suite evals

The 9 report-only disruption skills need a report_judge verifier: schema-conformance + a zero-action policy gate (calling cancel/modify = instant fail) + judged severity correctness. New taskgen domain.

Stale re-runs + thin edges

Re-measure ancillery, disruption, hotel-search, flight-search on the 50-task banks. Split edge-* generation by family (edge-availability still has 1 dedicated task).
