2. Two-Repo Architecture

Knowledge and judgment live in separate repositories with a narrow contract: the skills repo ships SKILL.md files; the playground evaluates them, optimizes them, and gates their PRs.

Detection suite is deliberately un-routed. The 9 disruption-detection sub-skills produce reports and never call action tools, so the router excludes them and the current tool-call verifiers cannot grade them (see §12 for their planned eval path). An explicit test enforces this exclusion.

3. Agent Runtime & Mock World

One LangGraph agent (gpt-4o-mini), two conditions per task. The mock MCP server gives every run the same deterministic world: identical parameters always return identical flights, hotels, and availability — so a score change always means a behavior change, never world noise.

flowchart TD T[Task instruction.md] --> COND{Condition} COND -->|no_skill| A1[Agent: harness only
base_system_prompt + tool descriptions] COND -->|with_skill| A2[Agent: harness + SKILL.md
injected into system prompt] A1 --> TOOLS[Tool loop against mock MCP
search_flights · search_hotels · check_availability
create/modify/cancel_booking · get_fare_rules
get_itinerary · validate_passenger · add_ancillary] A2 --> TOOLS TOOLS --> RESP[Final response + recorded tool calls] RESP --> VER[Verifier scores the run] VER --> DB[(trajectory.db
every tool sequence persisted)]

Harness — the second optimizable layer

  • base_system_prompt — global agent behavior
  • tool_descriptions.* — per-tool guidance (e.g. the hotel/flight validate_passenger split)
  • Whitelisted keys only; optimizer edits one leaf at a time so YAML can never corrupt

Why a seeded mock world

  • Same params → same response, forever (hash-seeded RNG)
  • Caught a real miscalibration: a task demanded booking a flight the world deterministically reports unavailable
  • CI boots it fresh per run; health-polled before evals start
EVALUATION

4. A/B Eval Pipeline

The core measurement: does the skill make the agent better than no skill at all? Every task runs N trials in both arms; the weighted delta and regression rate feed the gate.

flowchart TD BANK[Task bank — tasks for this skill
each task.toml: verifier, weight, expected tools/params] --> TRIALS[ab_compare.py
N trials x 2 arms per task, concurrency-capped] TRIALS --> NS[no_skill scores] TRIALS --> WS[with_skill scores] NS --> DELTA[per-task delta = avg with_skill - avg no_skill
+ latency delta + token cost delta] WS --> DELTA DELTA --> AGG[weighted delta across bank
weight 3.0 = business-critical task] AGG --> GATE2{Tiered gate — §6} GATE2 --> OUT[(results/skill_ab_results.json
summary + full per-task detail)]
Cost reality: a full 50-task, 5-trial CI gate run = ~500 calls ≈ $0.03 and ~4 minutes on a free GitHub runner. The with-skill arm uses ~2× the input tokens of no-skill — the price of the skill prompt, also tracked per run.

5. Verifiers & Judge Hardening

Two verifier types. The LLM judge was a silent regression-inflater until June 10: any parse failure scored as 0.0 and averaged in. It now retries, salvages truncated JSON, and excludes infra errors from the mean.

flowchart TD R[Agent run output] --> VT{verifier type} VT -->|tool_call_check| TC[Required tools present?
Required params present?] TC -->|all present| S1[score 1.0] TC -->|missing| S0[score 0.0 + reason] VT -->|llm_judge| J[3 judge votes — gpt-4o-mini
512-token cap, short-reasoning instruction] J --> P{JSON parses?} P -->|yes| OKV[vote counted] P -->|truncated| SALV[salvage score field by regex
reasoning marked truncated] SALV --> OKV P -->|unparseable| RET{retry once} RET -->|recovers| OKV RET -->|fails again| EXC[vote EXCLUDED from mean
noted in reason] OKV --> MEAN[mean of valid votes = score] EXC --> ALL{all votes failed?} ALL -->|yes| INFRA[JUDGE_INFRA_FAILURE
flagged loudly — not scored as agent failure]
Why this mattered: 3 of planning-skill's 8 "regressions" contained literal judge error: Unterminated string — eval-infra noise read as skill failure. After the fix (and one repaired task), planning-skill flipped from −0.045 to +0.044.

6. The Tiered Gate

The decision logic that turns an A/B run into PASS / SOFT_BLOCK / BLOCK. Thresholds were recalibrated for N=3 trial noise; tier-1 protects business-critical tasks individually.

flowchart TD IN[A/B summary
weighted delta + regression rate + per-task deltas] --> T1{Tier 1 — critical
any critical-domain task delta ≤ -0.30?
or regression rate > 50%?} T1 -->|yes| B[BLOCK — do not merge] T1 -->|no| T2{Tier 2 — soft
weighted delta below soft floor?
or regression rate > 35%?} T2 -->|yes| SB[SOFT_BLOCK — investigate before merging] T2 -->|no| P2[PASS]
Known sharp edge: tier-1 triggers on a single critical task, so with ~12 weight-3.0 tasks per domain one consistent failure blocks the skill — that's intentional (it caught the contradictory booking tasks in §10), but it means each new critical task is effectively a unit test for the skill.

9. Bandit & Strategy Arms

A Thompson-sampling bandit learns which edit strategy works on which layer. Crashed or timed-out runs count as losses, so self-destructive strategies get deprioritized automatically.

11. Case Study — booking-skill, three rounds to PASS

Why per-task evidence beats aggregate scores: two critical tasks demanded opposite defaults, and each one-sided fix flipped whichever task it disfavored. The distinguishing signal — hotel vs flight — only emerged from reading the traces.

sequenceDiagram participant G as A/B Gate participant D as Diagnosis participant S as Skill + Harness G->>D: BLOCK — booking-flow-102 (hotel, name+dates)
verifier demands immediate create_booking D->>S: contact email/phone: ask-but-proceed, not hard-required S->>G: re-run G->>D: BLOCK — booking-flow-109 (flight, no passport)
judge demands ASKING for the passport D->>S: ask-first sentence in validate_passenger S->>G: re-run G->>D: BLOCK — 102 AND 109 together
the two tasks punish opposite defaults D->>S: split rule: hotels book directly with name+dates,
flights ask once for a missing passport S->>G: re-run G-->>D: PASS — +0.001, 2% regressions, zero flagged
The recurring failure signature across every repaired skill (planning, booking, modify-booking, fare-rules): "gather/confirm before acting" guidance colliding with single-turn verifiers. Cure pattern: an explicit, fully-specified instruction executes in one turn; clarification is reserved for genuinely ambiguous requests. fare-rules added the inverse rule too — a rules question must never execute the action it asks about (it was cancelling bookings when asked about penalties).
